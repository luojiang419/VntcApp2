# 009 便携包服务自启动与监听检测修复完成

## 本轮目标

- 按用户真实使用路径复测便携包
- 修复“启动便携版 `vnt_app.exe` 后，`vntcrustdesk` 没有同步启动，页面仍显示服务未运行、49999 未监听”的问题
- 让 `VNTC` 客户端在非管理员普通启动下，也能优先把已安装的 `vntcrustdesk` 服务拉起并正确检测监听状态

## 已完成内容

### 1. 已完成本轮备份

- 已创建备份：
  - `backup\014_便携包服务查询诊断与PowerShell直连修复前备份_20260601_111921`

### 2. 已修复便携包静默拉起服务的核心问题

- 已增强 `lib/remote_assist/remote_assist_health_service.dart`
- `warmUpBackgroundSilently()` 现在会输出更细的静默预热日志，便于区分：
  - 未找到可执行文件
  - 服务存在但未启动
  - 服务启动后仍未就绪
  - 直接拉起监听进程后仍未监听
- `_ensureServiceRunning()` 现在会记录：
  - 启动前服务检测结果
  - `Start-Service` 执行结果
  - 启动后服务状态
- `_queryService()` 现在改为三重兜底查询：
  1. `Get-Service`
  2. `Get-CimInstance Win32_Service`
  3. `sc.exe query`
- 这样即使某一条查询链在便携包环境里表现不稳定，也不会再把已存在的服务误判成“不存在”

### 3. 已修复 PowerShell 调用方式差异

- 已修改：
  - `lib/remote_assist/remote_assist_health_service.dart`
  - `lib/remote_assist/remote_assist_launcher.dart`
- 两处 `_runPowerShell(...)` 都不再走 `runInShell: true`
- 现在统一直连：
  - `C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe`
- 这一步是为了消除便携包环境下 shell 包装层带来的参数解释差异

### 4. 已完成便携包真实启动路径复测

- 已重新导出便携包：
  - `dist\portable\VNT_App_2.2_Windows_Portable.zip`
- 已按真实用户路径复测：
  1. 先关闭 `vnt_app` 与 `vntcrustdesk` 进程
  2. 手动停止 `vntcrustdesk` 服务
  3. 清空便携包 `logs\remote_assist.log`
  4. 直接启动：
     - `dist\portable\VNT_App_2.2_Windows_Portable\vnt_app.exe`
  5. 等待客户端静默拉起远程协助后台

- 本轮实测结果已确认：
  - 服务从 `Stopped` 变为 `Running`
  - 产生 `--service` 进程
  - 产生 `--server` 进程
  - `49999` 端口进入监听

### 5. 已确认客户端日志命中正确修复链路

- 便携包日志文件：
  - `dist\portable\VNT_App_2.2_Windows_Portable\logs\remote_assist.log`

- 这说明当前便携包已经不是之前那种“只会重复尝试 `--server`，却从不真正启动服务”的错误路径
