## 已确认的根因

### 1. `approve-mode` 被 `OVERWRITE_SETTINGS` 强制固定为 `click`
- 文件：
  - `D:\Myproject\vnt2.0\vntcrustdesk-src\src\vntc.rs`
- 当前现状：
  - `apply_profile()` 中有：
    - `insert_setting(&mut overwrite, keys::OPTION_APPROVE_MODE, "click");`
- 这会导致：
  - `Config::get_option("approve-mode")` 永远优先返回覆盖值 `click`
  - 即使 Flutter / UI / CLI 后续调用 `set_option("approve-mode", "password")`，也不会真正生效
  - 连接端最终仍走“等待对方点击接受”的分支

### 2. 配置系统已确认“覆盖值优先于持久化配置，并且覆盖值不可保存”
- 文件：
  - `D:\Myproject\vnt2.0\vntcrustdesk-src\libs\hbb_common\src\config.rs`
- 已确认逻辑：
  - `Config::get_option(k)` 的优先级为：
    - `OVERWRITE_SETTINGS`
    - 持久化配置 `CONFIG2.options`
    - `DEFAULT_SETTINGS`
  - `is_option_can_save(...)` 中，如果目标 key 已存在于 `OVERWRITE_SETTINGS`，则返回 `false`
  - 这意味着：
    - `approve-mode` 只要还在 `OVERWRITE_SETTINGS` 中，用户设置 `password` 就永远不可能成为真实生效值

### 3. Flutter 桌面初始化链路没有补齐 VNTC profile
- 文件：
  - `D:\Myproject\vnt2.0\vntcrustdesk-src\src\flutter_ffi.rs`
- 当前现状：
  - `initialize(app_dir, custom_client_config)` 里：
    - Android / iOS 会走 `crate::common::global_init()`
    - 桌面端不会走 `global_init()`
    - 桌面端当前只做了 `init_log(false, "flutter_ffi")`
- 影响：
  - 桌面端通过 Flutter FFI 进行密码/选项读写时，没有显式执行 `crate::vntc::apply_profile()`
  - `APP_NAME / DEFAULT_SETTINGS / OVERWRITE_SETTINGS / 其他 VNTC profile 规则` 可能与后台服务运行态不完全一致
- 额外确认：
  - `D:\Myproject\vnt2.0\vntcrustdesk-src\src\common.rs`
  - `global_init()` 当前本体非常轻，只做了：
    - `crate::vntc::apply_profile();`
    - Linux 下额外 Wayland 初始化
  - 所以本轮按原计划只在桌面 FFI 初始化中补 `crate::vntc::apply_profile()`，属于低风险修复。

### 4. CLI 配置密码入口已经存在，但还需要更明确的“业务生效校验”
- 文件：
  - `D:\Myproject\vnt2.0\vntcrustdesk-src\flutter\lib\main.dart`
- 当前现状：
  - 已存在 `_handleAccessPasswordConfiguration(List<String> args)`。
  - 已具备以下逻辑：
    - `bind.mainSetPermanentPassword(password: password)`
    - `bind.mainSetOption(key: kOptionVerificationMethod, value: 'use-permanent-password')`
    - `bind.mainSetOption(key: kOptionApproveMode, value: password.isEmpty ? 'click' : 'password')`
    - 300ms 后读回：
      - `mainGetPermanentPassword()`
      - `mainGetOption(kOptionVerificationMethod)`
      - `mainGetOption(kOptionApproveMode)`
    - 不一致时 `exit(1)`
- 现阶段判断：
  - 这段逻辑已经比最初设想更接近目标，但在 `approve-mode` 被覆盖为 `click` 的前提下，非空密码场景仍会失败。
  - 另外，清空密码时目前仍把 `verification-method` 写成 `use-permanent-password`，这一点建议在正式修复时再确认是否要同步切回更合理的模式。
