# 006 远程协助状态误判与强制修复增强

## 本轮目标

修复安装完成后，应用内“远程协助”页面仍误报：

- `服务未运行`
- `49999 未监听`
- `防火墙规则缺失`

并补强“已安装但服务/文件损坏”时的应用内修复链路，确保点击“安装/修复”能够真正恢复 `vntcrustdesk`。

## 已完成内容

### 1. 定位到页面状态误判根因

- 已复现并确认当前机器上存在以下现象：
  - 页面报 `服务未运行 / 49999 未监听 / 防火墙规则缺失`
  - 但系统底层实际可能已存在部分有效状态
- 已直接执行 `RemoteAssistHealthService` 内部对应的 PowerShell 片段
- 已确认根因之一是：
  - `lib/remote_assist/remote_assist_health_service.dart` 中服务检测与防火墙检测脚本的 JSON 管道写法有语法错误
  - 原脚本会报：`An empty pipe element is not allowed`
- 这会导致：
  - `_queryService()` 恒定拿到空结果
  - `_queryFirewallRule()` 恒定拿到空结果
  - 页面因此把服务/防火墙状态错误显示为异常

### 2. 修复页面状态检测

- 已修正 `RemoteAssistHealthService` 中的 PowerShell JSON 输出方式
- 现在服务检测与防火墙检测会先写入 `$result`，再统一 `ConvertTo-Json`
- 已补充失败日志：
  - 当 PowerShell JSON 查询 exit code 非 0 时，会把 stderr 写入 `remote_assist.log`

### 3. 修复 49999 监听误判

- 原逻辑只要本机任意进程监听 `49999` 就会判定为“监听中”
- 在当前机器上，官方 `RustDesk.exe` 也监听了同端口，因此旧逻辑存在误判风险
- 已改为：
  - 仅当 `vntcrustdesk.exe`
  - 且 `ExecutablePath` 与当前远程协助运行时路径一致
  - 且该进程确实监听 `49999`
  - 才判定为“49999 监听中”

### 4. 修复“已安装但修复不彻底”问题

- 本轮实测发现第二个真实问题：
  - 当 `VNTC RustDesk` 已登记在 MSI 中
  - 但服务或可执行文件已经损坏/缺失时
  - 仅执行 `msiexec /i` 的重新配置不足以恢复服务
- 已增强 `scripts/bootstrap_vntcrustdesk.ps1`
  - 新增对“已安装但运行时不完整”的识别
  - 当检测到服务缺失或核心可执行文件缺失时：
    1. 先静默卸载当前 `VNTC RustDesk`
    2. 再执行全新安装
- 这样“安装/修复”按钮现在不仅能补装，也能把损坏安装重新拉回健康状态

### 5. 坏状态机器实测通过

- 本轮直接利用当前机器的坏状态进行修复验证：
  - 当时 `vntcrustdesk` 服务不存在
  - `C:\Program Files\VNTC RustDesk\vntcrustdesk.exe` 也不存在
  - 页面误判仍可复现
- 修改后再次执行：
  - `scripts/bootstrap_vntcrustdesk.ps1`
- 已成功恢复：
  - `vntcrustdesk` 服务为 `Running`
  - `vntcrustdesk.exe` 已恢复
  - `49999` 由 `vntcrustdesk.exe` 自身监听
  - 防火墙规则存在且启用

### 6. 新产物已重新导出并重装验证
